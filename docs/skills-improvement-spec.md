# Спецификация: улучшение библиотеки навыков llm-skills

**Дата:** 2026-07-02
**Статус:** черновик на утверждение
**Основание:** [аудит 2026-07](skills-audit-2026-07.md)

## 1. Резюме (TL;DR)

Библиотека в хорошем состоянии (7 навыков — OK, 1 — переработка), но несёт три системные проблемы: 16 из 28 описаний превышают лимит Anthropic в 1024 символа и создают токен-налог ~15–18 тыс. токенов на каждую сессию; триггер agent-audit конфликтует со всем семейством аудиторов; django-tailwind-optimizer учит устаревший (и внутренне сломанный) Tailwind v3-workflow. Спека фиксирует решения по пяти категориям находок и четыре ADR-вопроса. Реализация — в [плане](skills-improvement-plan.md), недостающие навыки — в [ROADMAP.md](../ROADMAP.md).

## 2. Проблема и контекст

### 2.1. Текущая ситуация

28 навыков, все загружаются в Claude Code как плагин. Метаданные всех навыков (~31 тыс. символов описаний) постоянно находятся в системном промте. Библиотека выросла итеративно: раннее поколение (500-error-eliminator, django-tailwind-optimizer, git-commit-planner, techlead-ai, fact-checker) имеет разнобойную структуру; позднее «семейство аудиторов» — единый выверенный скелет. Анти-коллизионные указатели добавлялись по мере обнаружения конфликтов и в основном работают.

### 2.2. Потребность

Владелец — соло-разработчик, платит за токены. Цели: (1) снизить фоновую токен-стоимость библиотеки; (2) исключить ложные срабатывания навыков; (3) не дать инструкциям устаревать молча; (4) закрыть пробелы жизненного цикла (отдельно — ROADMAP).

## 3. Цели и анти-цели

### 3.1. В scope

- Все описания ≤ 1024 символов с сохранением триггеров и анти-коллизионных указателей.
- Дефуз конфликтных триггеров (agent-audit, python-project-audit).
- Актуализация django-tailwind-optimizer до Tailwind v4.
- README и структурная унификация (references/, версии).
- ROADMAP.md с кандидатами в новые навыки.

### 3.2. Анти-цели

- НЕ переписывать работающие тела навыков ради стиля (7 OK-навыков не трогаем).
- НЕ переводить тела навыков на английский (см. ADR-1).
- НЕ вводить обязательные evals для всех навыков — только адресно для спорных пар и только по явному решению.
- НЕ менять имена навыков — они в маршрутизации глобального CLAUDE.md и в памяти сессий.

## 4. Решения по категориям

### 4.1. Описания (категория A1 аудита)

Правило рерайта для 16 нарушителей — целевая структура ≤ 1024 символов (в идеале 600–800):

1. Одно предложение «что делает» (без перечисления половины тела — этим грешат fastapi-architect, goal-pipeline).
2. 3–6 самых сильных триггерных фраз вместо 10–15 (длинные списки near-синонимов «проверь проект», «аудит проекта», «оцени мой бэкенд» сжимаются до 2–3 представителей — модель обобщает сама).
3. Анти-коллизионный указатель — сохранить обязательно, сжать до формулы «Для X — см. Y».
4. Ограничение области («только aiogram 3.x», «не для единичных файлов») — сохранить.

Проверка: скрипт замера длины (однострочник из аудита) прогоняется по всем 28 после правок; ни одного значения > 1024.

### 4.2. Триггеры и коллизии (A2)

- **agent-audit:** убрать одиночные «аудит»/«audit» из триггеров; оставить контекстные формы («проверь себя», «самоаудит», «что ты упускаешь», «мета-ревью») + явное «НЕ для аудита кода/проекта — см. django-audit / python-project-audit».
- **python-project-audit:** убрать «найди баги» (уводит от /code-review и techlead-ai) и «проверь безопасность проекта» (уводит от /security-review и django-audit security); оставить ядро «production readiness / проверка на блуд / балльная оценка».
- **django-audit ↔ test-coverage-auditor:** добавить взаимные указатели (линза tests — обзорная в рамках Django-аудита; глубокий разбор качества тестов — test-coverage-auditor).
- **500-error-eliminator:** добавить указатель на vps-deploy-auditor для инфраструктурных 500 (nginx/gunicorn/systemd).

### 4.3. Устаревшее содержание (A3)

**django-tailwind-optimizer** — переработка на v4-first:

- Основной путь: Tailwind v4 (standalone CLI v4, `@import "tailwindcss"`, `@theme` в CSS, без `tailwind.config.js`).
- Раздел «Legacy: проект на v3» — сохранить текущие инструкции как ветку для существующих проектов, с явным определением версии на шаге анализа.
- Зафиксировать версию в командах установки (не `latest`), чтобы инструкция не ломалась молча при следующем мажоре.
- 264 строки без references: вынести пошаговые миграции в `references/migration.md`.

Правило впредь: любой навык, привязанный к версии инструмента, обязан (а) называть версию в первом абзаце тела, (б) определять фактическую версию проекта до выдачи рекомендаций.

### 4.4. Структура и консистентность (A4, A5, A6)

- `REFERENCE.md` → `references/reference.md` в session-catchup и windows-pwsh-terminal (пути в телах обновить).
- `ratchet-loop/contract-example.md` → `ratchet-loop/references/contract-example.md`.
- README: добавить `ratchet-loop/` и `.claude-plugin/` в дерево; добавить «Вариант 0» установки — через plugin marketplace.
- Тяжеловесы: goal-pipeline — вынести «Протокол исполнителя», «Маркеры транскрипта», «Память (Windows)» в `references/executor-protocol.md`; ratchet-loop — аналогично («Маркеры транскрипта», «Память (Windows)»); spec-writer — вынести три полных шаблона в `references/templates.md`, в теле оставить каркасы разделов. Целевой размер тела: ≤ 12 КБ.
- Имена справочников формата вывода: новые файлы — только `output-format.md`; существующие `report-template.md`/`report-format.md` не переименовывать (низкая ценность, риск битых ссылок), но зафиксировать конвенцию в README.

### 4.5. Процесс сопровождения (A7)

- Бампать `version` в `plugin.json` при каждом содержательном изменении навыков (иначе кэш маркетплейса не обновится); `metadata.version` навыка — при изменении этого навыка.
- Языковая унификация заголовков — попутно при правке конкретного навыка, не отдельной кампанией.

## 5. ADR-вопросы

### ADR-1. Язык описаний: русский vs английский

**Контекст:** кириллица ≈ ×2 токенов; описания сидят в каждой сессии. 28 описаний на английском стоили бы ~2× дешевле. При этом библиотека личная, триггеры на русские формулировки владельца работают, README декларирует русский как фичу.

- **Вариант A (рекомендую):** оставить русский, но сжать до 600–800 символов — экономия ~40–50 % без смены политики.
- **Вариант B:** английские описания с русскими триггер-цитатами («говорит „проверка на блуд“») — ещё ~30 % экономии, риск ухудшения триггеринга на русских запросах, проверять только evals.
- **Решение:** A сейчас; B рассмотреть, только если после сжатия токен-налог всё ещё мешает.

### ADR-2. Судьба сверхдлинных «энциклопедических» описаний

**Контекст:** goal-pipeline/ratchet-loop сознательно несут в описании развёрнутую анти-коллизию (НЕ ralph-loop, НЕ gate-loop...). Сжатие может ослабить разграничение.

- **Вариант A (рекомендую):** анти-коллизия остаётся в описании в сжатой форме (одна строка), развёрнутая таблица «Чем НЕ является» — уже есть в теле, там ей и место.
- **Вариант B:** оставить длинные описания как осознанное исключение. Против: против него и лимит, и токен-налог.

### ADR-3. Evals для спорных пар

**Контекст:** коллизии чинились «на глаз»; skill-creator даёт эмпирическую проверку, но каждый прогон — токено-затратный.

- **Решение:** опциональная фаза. Кандидаты в порядке ценности: (1) agent-audit vs семейство аудиторов — самая рискованная пара; (2) django-audit vs python-project-audit; (3) docs-generator vs spec-writer. Запуск — только по явной команде владельца, по одной паре за раз.

### ADR-4. `metadata.version` во фронтматтере

**Контекст:** поле нестандартное (Anthropic его не читает), но дисциплинирует и позволяет отслеживать эволюцию навыка.

- **Решение:** оставить; считать источником правды при бампе версии плагина (плагин бампается, если бампнулся хотя бы один навык).

## 6. Риски и смягчение

| Риск | Вероятность | Смягчение |
|---|---|---|
| Сжатие описания ухудшит триггеринг конкретного навыка | средняя | менять формулировки, а не смыслы: сильные триггер-фразы сохраняются дословно; спорные пары — ADR-3 (evals) |
| Перенос разделов в references сломает внутренние ссылки | низкая | после каждого переноса — grep по имени файла по всему репо |
| Лимит 1024 — неточно процитирован | низкая | помечено «требует проверки»; даже если лимита нет, токен-мотивация сжатия остаётся |
| Правки разъедутся с кэшем плагина | средняя | бамп `plugin.json` + переустановка плагина после мержа |
