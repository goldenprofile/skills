# План реализации: улучшение библиотеки llm-skills

**Дата:** 2026-07-02
**Основание:** [спека](skills-improvement-spec.md), [аудит](skills-audit-2026-07.md)
**Исполнитель:** Claude Code под присмотром владельца; фазы независимы и коммитятся отдельно.

## 1. Обзор

Пять фаз по убыванию (ценность / усилие). Фазы 1–2 закрывают все high-находки аудита. Каждая фаза — один сеанс работы и один-два атомарных коммита. Фаза 5 — опциональная и токено-затратная, по явной команде.

## 2. Фазы

### Фаза 1: Быстрые победы — гигиена репозитория (оценка: ~30 мин)

Без содержательных правок навыков; риск минимальный.

1. README: добавить `ratchet-loop/` и `.claude-plugin/` в дерево структуры; добавить «Вариант 0: установка как плагина» (`/plugin marketplace add goldenprofile/llm-skills`); зафиксировать конвенцию имён справочников (`output-format.md` для новых).
2. `session-catchup/REFERENCE.md` → `session-catchup/references/reference.md`; обновить ссылку в теле.
3. `windows-pwsh-terminal/REFERENCE.md` → `windows-pwsh-terminal/references/reference.md`; обновить ссылку.
4. `ratchet-loop/contract-example.md` → `ratchet-loop/references/contract-example.md`; обновить ссылку.
5. Grep по репо на битые ссылки после переносов.

**Выход:** коммит `docs+chore`. Зависимостей нет.

### Фаза 2: Описания и триггеры (оценка: ~2 ч)

Самая ценная фаза: закрывает A1 (лимит/токен-налог) и A2 (коллизии).

1. Переписать 16 описаний > 1024 символов по правилу из спеки §4.1 (цель 600–800, сохранить сильные триггеры и анти-коллизию). Порядок — от худших: goal-pipeline (1980), python-project-audit (1885), ratchet-loop (1873), spec-writer (1544), fastapi-architect (1452), claude-code-auditor (1399), vps-deploy-auditor (1315), django-audit (1292), docs-generator (1261), dependency-auditor (1216), migration-safety-auditor (1141), agent-audit (1081), harness-engineering (1062), techlead-ai (1056), obsidian (1035), clarify-prompt (1028).
2. Дефуз триггеров: agent-audit (убрать одиночное «аудит», добавить «НЕ для аудита кода — см. ...»); python-project-audit (убрать «найди баги», «проверь безопасность проекта»).
3. Добавить взаимные указатели: django-audit ↔ test-coverage-auditor; 500-error-eliminator → vps-deploy-auditor.
4. Прогнать скрипт замера длины по всем 28 — ни одного > 1024.
5. Бампнуть `metadata.version` затронутых навыков (minor) и `plugin.json` → 1.1.0.

**Выход:** коммит `feat(descriptions)`. Зависимости: нет (фаза 1 не обязательна).
**Контрольная точка:** показать владельцу до/после для трёх самых спорных (agent-audit, python-project-audit, goal-pipeline) перед коммитом.

### Фаза 3: Содержательные переработки (оценка: ~2–3 ч)

1. **django-tailwind-optimizer → v4-first** (спека §4.3): основной путь v4 (CSS-first, `@theme`, standalone CLI v4 с зафиксированной версией), ветка «Legacy v3» для существующих проектов, обязательный шаг определения версии; миграционные планы — в `references/migration.md`. Перед правкой свериться с актуальной докой Tailwind (WebFetch) — детали v4-конфига в аудите помечены «требует проверки».
2. **Разгрузка тяжеловесов** (спека §4.4): goal-pipeline — протокол исполнителя/маркеры/память → `references/executor-protocol.md`; ratchet-loop — маркеры/память → `references/loop-protocol.md`; spec-writer — три шаблона → `references/templates.md`. Целевой размер тел ≤ 12 КБ; в теле остаются каркасы и указатели.
3. Попутная языковая унификация заголовков в правленых файлах (500-error-eliminator — если руки дойдут, отдельным мелким коммитом).
4. `metadata.version` затронутых навыков; `plugin.json` → 1.2.0.

**Выход:** 2–3 коммита по навыкам. Зависимость: фаза 2 желательна раньше (чтобы не переписывать описания дважды).
**Контрольная точка:** django-tailwind-optimizer — прогнать сгенерированные команды v4 на тестовой папке до коммита.

### Фаза 4: Новые навыки по ROADMAP (оценка: ~1 сеанс на навык)

Порядок — по приоритетам [ROADMAP.md](../ROADMAP.md): сначала test-writer и observability-bootstrap (high). Каждый навык:

1. Скелет по образцу «семейства аудиторов» (Когда применять → Контекст → Процесс → Уровни риска → Чеклист → Связь с библиотекой → Справочники).
2. Описание сразу ≤ 1024 символов, с анти-коллизией к соседям.
3. Обновить README-каталог и `plugin.json` version.
4. Проверка на реальной задаче до коммита (правило README §«Создание нового навыка»).

**Зависимость:** фаза 2 (чтобы новые описания писались по уже принятой норме).

### Фаза 5 (опциональная, токено-затратная): эмпирическая проверка триггеров

Только по явной команде владельца, по одной паре за прогон, через evals skill-creator (`evals/evals.json` + benchmark):

1. agent-audit vs django-audit/python-project-audit — 6–8 промтов («сделай аудит проекта», «проверь себя», «что устарело» ...), проверка какой навык триггерится.
2. django-audit vs python-project-audit.
3. docs-generator vs spec-writer.

**Вход:** фаза 2 завершена (иначе меряем старые описания). **Выход:** при провале пары — итерация описаний с повторным прогоном.

## 5. Риски и внешние зависимости

- Точный синтаксис Tailwind v4 — сверить по официальной доке в момент выполнения фазы 3 (внешняя зависимость).
- После фаз 2–4 переустановить плагин из маркетплейса, иначе сессии продолжат грузить старый кэш.
- Формулировки триггеров проверяются вживую в следующих рабочих сессиях; регресс триггеринга — откат конкретного описания (изменения атомарны по навыку).
