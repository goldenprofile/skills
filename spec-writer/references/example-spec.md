# Система email-уведомлений — Техническая спецификация

> **Статус:** Draft
> **Дата:** 2026-06-23
> **Автор:** Ivan

---

## 1. Резюме (TL;DR)

Добавляем систему транзакционных email-уведомлений в проект. Пользователи будут
получать письма при ключевых событиях (регистрация, смена пароля, новый платёж).
Уведомления отправляются асинхронно через Celery, шаблоны писем рендерятся через
Django Templates с поддержкой HTML + plain-text. Отказоустойчивость через retry
с exponential backoff.

## 2. Проблема и контекст

### 2.1. Текущая ситуация

Сейчас уведомления отправляются только через Telegram-бота (синхронно, в момент
события). Email-канал отсутствует полностью. Это проблема: часть пользователей
не использует Telegram, а email — универсальный канал, ожидаемый в B2B-сегменте.

### 2.2. Бизнес-потребность

- **Охват:** ~30% пользователей не подключены к Telegram → теряем их как
  информированных пользователей.
- **Доверие:** Email-уведомления — индустриальный стандарт для транзакционных
  писем. Их отсутствие вызывает вопросы у enterprise-клиентов.
- **Конверсия:** Письма о новых возможностях → возврат пользователей в продукт.

## 3. Цели и анти-цели

### 3.1. Что в scope
- Транзакционные email-уведомления для 5 событий: регистрация, подтверждение email,
  сброс пароля, новый платёж, изменение тарифа.
- Асинхронная отправка через Celery.
- HTML + plain-text версии каждого письма.
- Retry при ошибках отправки (3 попытки с exponential backoff).
- Логирование отправки и ошибок.

### 3.2. Что НЕ в scope (анти-цели)
- Массовые рассылки / маркетинговые кампании (это отдельная система).
- Редактор шаблонов в админке (пока шаблоны — файлы в репозитории).
- Сегментация пользователей для рассылок.
- Трекинг открытий и кликов (будет позже, когда понадобится).

## 4. Предлагаемое решение

### 4.1. Обзор архитектуры

```
Django Signal (post_save, etc.)
    │
    ▼
Celery Task (send_transactional_email)
    │
    ├── Рендерит HTML + plain-text из Django Templates
    │
    ▼
Email Backend (Anymail → Mailgun / SMTP)
    │
    ├── Успех → лог: EmailLog(status=SENT)
    └── Ошибка → retry(max_retries=3, countdown=2**retry * 60)
                 → после 3 ошибок: лог: EmailLog(status=FAILED)
```

### 4.2. Ключевые проектные решения

Для каждого значимого решения — формат ADR-lite:

**Решение 1: Anymail вместо прямого SMTP**
- **Контекст:** Нужен провайдер транзакционных писем с хорошей доставляемостью.
- **Варианты:** (a) Anymail + Mailgun, (b) django-ses + AWS SES, (c) прямой SMTP
- **Выбор:** Anymail + Mailgun
- **Обоснование:** Anymail даёт единый API для разных провайдеров, не вендор-лок.
  Mailgun выбран потому что проект уже использует Mailgun-домен для других целей.
- **Последствия:** упрощается смена провайдера в будущем; появляется зависимость
  от Anymail (но это стабильный пакет с 2K+ звёзд).

**Решение 2: Django Templates для писем вместо специализированного MJML**
- **Контекст:** Нужны HTML-письма, совместимые с основными почтовыми клиентами.
- **Варианты:** (a) Django Templates с инлайн-стилями, (b) MJML с компиляцией
- **Выбор:** Django Templates
- **Обоснование:** Не добавляем новый инструмент в стек ради 5 писем. Инлайн-стили
  работают для простых транзакционных писем. MJML — избыточен на этом этапе.
- **Последствия:** письма могут выглядеть чуть хуже в Outlook (но это приемлемо
  для B2B-аудитории, где Outlook — редкость).

**Решение 3: Celery для асинхронности**
- **Контекст:** Отправка письма не должна блокировать HTTP-ответ.
- **Варианты:** (a) Celery, (b) django-background-tasks (БД-бэкенд), (c) потоки
- **Выбор:** Celery (уже есть в проекте)
- **Обоснование:** Проект уже использует Celery для других задач. Не добавляем
  новую инфраструктуру.

### 4.3. Модель данных

```python
class EmailLog(models.Model):
    """Лог отправки каждого письма."""

    user = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    event_type = models.CharField(max_length=50)  # 'registration', 'password_reset', ...
    recipient = models.EmailField()
    subject = models.CharField(max_length=200)
    status = models.CharField(
        max_length=20,
        choices=[("pending", "Pending"), ("sent", "Sent"), ("failed", "Failed")],
        default="pending",
    )
    provider_message_id = models.CharField(max_length=255, blank=True)
    error_message = models.TextField(blank=True)
    retry_count = models.PositiveSmallIntegerField(default=0)
    created_at = models.DateTimeField(auto_now_add=True)
    sent_at = models.DateTimeField(null=True, blank=True)
```

## 5. Альтернативы (отклонённые)

- **Django's built-in `send_mail` + SMTP напрямую:** Отклонено — нет retry,
  нет провайдер-абстракции, сложнее мониторинг доставляемости.
- **Сторонний сервис рассылок (SendGrid, Unisender):** Отклонено — это
  транзакционные письма, а не рассылки. Mailgun покрывает этот сценарий.
- **WebSocket + фронтенд-уведомления вместо email:** Отклонено — это
  дополнительный канал, а не замена email.

## 6. Риски и смягчение

| Риск | Вероятность | Влияние | Смягчение |
|------|------------|---------|-----------|
| Письма попадают в спам | Средняя | Высокое | Настроить SPF/DKIM/DMARC на домене до запуска; мониторить репутацию через Mailgun dashboard |
| Mailgun downtime | Низкая | Среднее | Retry с backoff покрывает короткие сбои (до 14 мин суммарно) |
| Ошибки в шаблонах (битый HTML) | Средняя | Низкое | Валидация HTML в тестах; plain-text фолбэк всегда отправляется |
| Рост очереди Celery при массовых событиях | Низкая | Среднее | Транзакционные письма — единичные события; не ожидается всплесков. Мониторинг длины очереди. |

- [ ] Все 5 типов писем отправляются на тестовый email
