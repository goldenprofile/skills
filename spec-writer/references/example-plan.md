# Система email-уведомлений — План реализации

> **На основе:** `2026-06-23-email-notifications-spec.md`
> **Дата:** 2026-06-23

---

## 1. Обзор

Реализация транзакционных email-уведомлений через Anymail + Mailgun с
асинхронной отправкой через Celery. Оценка: 3-4 дня при работе solo.

## 2. Фазы реализации

### Фаза 1: Инфраструктура и модель (оценка: 4-6 часов)

**Цель фазы:** Всё что нужно до написания первого письма: пакеты, настройки, модель.

**Задачи:**
- [ ] Задача 1.1 — Установить `django-anymail` + добавить настройки Mailgun в
  `settings.py` (API key через переменные окружения, не хардкодить).
- [ ] Задача 1.2 — Создать `EmailLog` модель + миграцию.
- [ ] Задача 1.3 — Добавить `EmailLog` в админку (readonly для логов).
- [ ] Задача 1.4 — Написать `send_transactional_email` Celery-таск с базовой
  логикой отправки (без retry пока).

**Зависимости:** нет (можно начинать сразу)
**Критерий завершения:** Таск отправляет тестовое письмо через Mailgun sandbox
domain, запись создаётся в `EmailLog`.

### Фаза 2: Шаблоны писем (оценка: 4-5 часов)

**Цель фазы:** HTML + plain-text шаблоны для всех 5 типов писем.

**Задачи:**
- [ ] Задача 2.1 — Создать базовый HTML-макет письма (header/footer/стили).
- [ ] Задача 2.2 — Написать шаблоны для 5 событий (registration, email_confirm,
  password_reset, new_payment, plan_change).
- [ ] Задача 2.3 — Plain-text версии для каждого письма.
- [ ] Задача 2.4 — Тесты на рендеринг шаблонов (HTML содержит ключевые строки,
  не битый).

**Зависимости:** Фаза 1 (нужны настройки Anymail для тестового рендеринга)
**Критерий завершения:** Все 5 шаблонов рендерятся, тесты проходят.

### Фаза 3: Интеграция с бизнес-логикой (оценка: 3-4 часа)

**Цель фазы:** Привязать отправку писем к реальным событиям в проекте.

**Задачи:**
- [ ] Задача 3.1 — Подключить signals/handlers для каждого из 5 событий.
- [ ] Задача 3.2 — Написать интеграционные тесты (событие → таск в очереди →
  письмо отправлено).
- [ ] Задача 3.3 — Добавить retry с exponential backoff в Celery-таск.

**Зависимости:** Фаза 2
**Критерий завершения:** При создании пользователя → welcome-письмо в очереди.
Тест проходит полный цикл для каждого типа события.

### Фаза 4: Отказоустойчивость и мониторинг (оценка: 2-3 часа)

**Цель фазы:** Письма не теряются, ошибки видны.

**Задачи:**
- [ ] Задача 4.1 — Настроить алерты в Sentry при статусе FAILED после 3 retry.
- [ ] Задача 4.2 — Добавить management command для ручной переотправки FAILED писем.
- [ ] Задача 4.3 — Настроить SPF/DKIM/DMARC на домене.

**Зависимости:** Фаза 3
**Критерий завершения:** FAILED-письмо → алерт в Sentry. `manage.py resend_failed_emails`
отправляет письма заново. DNS-записи проходят валидацию.

## 3. График зависимостей

```
Фаза 1 (инфраструктура)
    │
    ▼
Фаза 2 (шаблоны)
    │
    ▼
Фаза 3 (интеграция)
    │
    ▼
Фаза 4 (отказоустойчивость)
```

Все фазы последовательные — каждая опирается на результат предыдущей.

## 5. Риски и зависимости

### Внутренние риски
- **Сложность HTML-вёрстки писем:** Письма — это не веб, нужны таблицы и
  инлайн-стили. Если 5 шаблонов окажутся сложнее ожидаемого → добавить +1 день.
  Смягчение: использовать проверенный базовый шаблон из `example-spec.md`.

### Внешние зависимости
- **Доступ к Mailgun-аккаунту:** Нужны API-ключи и настройка домена.
  От кого завишу: админ проекта.
- **DNS-записи (SPF/DKIM/DMARC):** Нужен доступ к DNS-зоне домена.
  От кого завишу: админ домена (может быть тот же человек).

## 6. Что НЕ входит в план

- Массовые рассылки / маркетинговые кампании
- Редактор шаблонов в админке
- Трекинг открытий и кликов
- A/B-тестирование тем писем
- Локализация писем (только русский язык)
