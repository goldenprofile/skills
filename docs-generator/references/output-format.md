# Формат вывода

Два режима: **аудит** (нашли отсутствующее/устаревшее) и **генерация** (создали/обновили доки).
В обоих — сначала вердикт, потом детали. Правки в файлы — по согласованию.

## Режим АУДИТА

Сначала вердикт, потом находки по убыванию приоритета.

```markdown
## Аудит документации — вердикт

**Состояние доков:** ПЛОХО / ЕСТЬ ПРОБЕЛЫ / ОК
**Проект:** FastAPI-сервис аналитики · Python 3.12 · systemd-деплой
**Проверено:** README, docs/adr/ (2 шт.), CLAUDE.md, AGENTS.md, docstrings в app/services/

Одной строкой: где документация врёт или чего критично не хватает.

---

## Находки

### CRITICAL — README указывает несуществующую команду запуска
- **Где:** `README.md`, секция «Быстрый старт»
- **Что не так:** `python main.py` — такого файла нет, точка входа `app/main.py` под uvicorn.
- **Почему важно:** агент/будущий ты не запустит проект; первая сессия начнётся с поломки.
- **Как исправить:** заменить на `uvicorn app.main:app --reload`; сверить с реальным кодом.

### HIGH — CLAUDE.md есть, AGENTS.md отсутствует
- **Где:** корень репозитория
- **Что не так:** OpenCode работает без правил проекта (читает AGENTS.md).
- **Как исправить:** создать AGENTS.md (symlink или указатель на CLAUDE.md — см. agent-docs.md).

### MEDIUM — выбор Postgres вместо SQLite нигде не зафиксирован
- **Где:** `docs/adr/` (отсутствует ADR)
- **Что не так:** нетривиальное решение без причины; через полгода вопрос откроется заново.
- **Как исправить:** завести `docs/adr/0001-use-postgres.md` по шаблону.

### LOW — docstring дублирует type hints
- **Где:** `app/services/timeout.py:set_timeout`
- **Как исправить:** убрать пересказ типов, оставить смысл/краевые случаи.

---

## Что предлагаю сгенерировать/исправить

Список конкретных правок с приоритетом. По согласованию — применяю (создаю/правлю файлы).
- [ ] Починить команду запуска в README (CRITICAL)
- [ ] Создать AGENTS.md как указатель на CLAUDE.md (HIGH)
- [ ] ADR-0001 о выборе Postgres (MEDIUM)
```

## Режим ГЕНЕРАЦИИ

Когда создаём/обновляем доки — отчёт о том, что сделано.

```markdown
## Документация — сгенерировано

**Проект:** <название/класс>
**Режим:** генерация (создано недостающее)

### Создано
- `README.md` — назначение, quick start, env, запуск, деплой-заметка systemd.
- `docs/adr/0001-use-postgres.md` — фиксация выбора БД.
- `AGENTS.md` — указатель на CLAUDE.md (синхронизация).

### Обновлено
- `app/services/payment.py` — добавлены docstrings к публичным функциям (Google-style).

### Требует твоего ввода (не вытащить из кода)
- ADR по выбору async vs sync — нужна твоя мотивация решения.
- Деплой-заметка: подтверди имя systemd-юнита и путь к EnvironmentFile.
```

## Принципы вывода

- **Вердикт первым.** «Документация врёт / есть пробелы / ок» — потом детали.
- **Приоритет по вреду:** документация, которая **врёт** (расходится с кодом) — всегда выше, чем
  отсутствие документации. Неверная команда запуска = CRITICAL; нет docstring = LOW/MEDIUM.
- **Конкретика:** файл, секция, что именно не так и как починить — не «улучшить документацию».
- **Не раздувай.** Для скрипта на 50 строк отсутствие ADR и docstrings — не находка. Привязывай
  требования к размеру и важности проекта (личная утилита ≠ боевой сервис).
- **Документируй «почему».** Любая правка следует принципу: объясняем причину, не пересказываем код.
- **Правки — по согласованию.** Сначала отчёт; создаём/правим файлы после подтверждения.
- **Незнакомый проект** — сперва `code-archaeologist`, иначе документируешь вслепую.
