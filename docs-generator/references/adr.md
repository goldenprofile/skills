# ADR — Architecture Decision Records (лёгкий формат)

ADR — короткая запись о **принятом архитектурном решении**: контекст, само решение и его
последствия. Цель для соло — через полгода вспомнить *почему* выбрал так, и не переоткрывать
уже закрытые вопросы. Это память «будущего тебя» и контекст для агента. Используем лёгкий формат
в духе **MADR** (Markdown Any Decision Records) — без тяжёлого процесса.

## Когда писать ADR (а когда нет)

**Писать**, если решение:
- трудно или дорого откатить (выбор БД, фреймворка, схемы аутентификации, формата API);
- неочевидно и кто-то (включая будущего тебя) спросит «почему так, а не иначе»;
- отвергает разумную альтернативу (выбрал Postgres вместо Mongo; sync вместо async; отказался
  от Celery в пользу systemd-timer);
- задаёт правило на весь проект (структура модулей, формат ошибок, стратегия миграций).

**Не писать** для: обратимых мелочей, выбора имени переменной, очевидных решений без альтернатив,
сиюминутных хаков (это — комментарий в коде или TODO).

> Правило большой пальца: «Буду ли я (или агент) помнить причину через 6 месяцев?» Нет → ADR.

## Где хранить и как нумеровать

- Каталог: **`docs/adr/`** в корне репозитория.
- Имя файла: `NNNN-kebab-case-title.md`, где `NNNN` — сквозной номер с нулями: `0001`, `0002`.
- Номер **не переиспользуется** и не меняется. Решение отменили — статус `Superseded`, файл живёт.
- Опционально `docs/adr/README.md` — индекс (список ADR со статусами); можно вести руками.

```
docs/adr/
├── 0001-use-postgres-not-sqlite.md
├── 0002-drop-celery-for-systemd-timers.md
├── 0003-fastapi-over-django-for-analytics-api.md
└── README.md          # индекс (опционально)
```

## Шаблон записи

```markdown
# NNNN. Краткое решение в повелительном/утвердительном виде

- **Статус:** Accepted        <!-- Proposed | Accepted | Deprecated | Superseded by ADR-0007 -->
- **Дата:** 2026-06-16

## Контекст

Какую проблему решаем и в каких условиях. Что вынуждает принять решение: требования,
ограничения (соло, Windows-dev, systemd-прод, бюджет времени), факты. Без «решения» здесь —
только постановка. Почему вопрос вообще встал.

## Решение

Что именно решили — одним-двумя абзацами, конкретно. «Используем Postgres как основную БД для
всех проектов, требующих конкурентной записи; SQLite — только для локальных утилит.»

## Альтернативы

Что рассматривали и почему отвергли (1–2 строки на вариант). Это самое ценное для будущего тебя.
- **SQLite** — отверг: нет нормальной конкурентной записи, прод на одном сервере, но многопоточно.
- **Mongo** — отверг: данные реляционные, не нужна гибкая схема, лишняя инфраструктура.

## Последствия

Что меняется в результате — и хорошее, и плохое. Честно про минусы.
- (+) Транзакции, FK, зрелый стек, уже есть на сервере.
- (−) Нужно поднимать/бэкапить Postgres локально на Windows-dev.
- Следствие: появляются Alembic-миграции → перед прод-деплоем гонять `migration-safety-auditor`.
```

## Статусы и жизненный цикл

- **Proposed** — рассматривается (для соло часто пропускается, сразу Accepted).
- **Accepted** — принято и действует.
- **Deprecated** — больше не актуально, замены нет.
- **Superseded by ADR-NNNN** — заменено другим решением. В новом ADR — ссылка `Supersedes ADR-NNNN`.

**Никогда не редактируй принятое решение задним числом** (кроме опечаток). Передумал — заводи
новый ADR, старый помечай Superseded. История решений важнее «чистоты».

## Связь с кодом и другими доками

- ADR объясняет «почему», ARCHITECTURE.md (см. `harness-engineering`) — «как устроено сейчас».
  Если они расходятся — ARCHITECTURE.md описывает текущее, ADR — историю выбора.
- В коде/PR, реализующем решение, можно сослаться: `# см. ADR-0002`.
- README ссылается на `docs/adr/` общей строкой, не пересказывает решения.

## Аудит ADR

- **HIGH:** принято крупное необратимое решение, ADR нет → причина потеряна.
- **MEDIUM:** ADR без раздела «Последствия» или «Альтернативы» (теряется главная ценность).
- **LOW:** нумерация с пропусками, нет индекса, статус не обновлён после замены решения.
