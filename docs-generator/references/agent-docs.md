# CLAUDE.md / AGENTS.md — документация для агентов

Это память проекта для AI-агентов. **Claude Code** читает `CLAUDE.md`, **OpenCode** (и ряд других
инструментов) — `AGENTS.md`. У владельца оба инструмента, поэтому **оба файла должны существовать
и быть синхронизированы**. Принцип жёсткий: **кратко и проверяемо**. Лишний контекст вредит —
агент тонет в воде и игнорирует важное.

> Граница ответственности: docs-generator поддерживает существование и синхронизацию этих файлов
> и «человеческую» документацию. **Глубокую** настройку агентской обвязки (DoD, tooling, canonical
> docs, вшивание вызовов навыков в Definition of Done) делает навык **`harness-engineering`** —
> при серьёзной настройке среды для агентов передавай управление ему.

## Что писать (и чего НЕ писать)

**Писать** — то, что агент не выведет сам и что проверяемо:
- как запускать/тестировать/линтить (конкретные команды, не «запусти тесты»);
- нерушимые правила и ограничения проекта («миграции только через `migration-safety-auditor`»,
  «secrets из env, не в коде», «не трогать `legacy/`»);
- неочевидные соглашения (структура модулей, формат ошибок, стиль коммитов);
- ссылки на canonical docs (ARCHITECTURE.md, ADR) — не пересказ, а указатель;
- Definition of Done, если он есть (что значит «готово»).

**НЕ писать:** пересказ кода, общие правила хорошего тона («пиши чистый код»), длинные объяснения,
то, что уже в README, неактуальные заметки. Если правило нельзя проверить — оно бесполезно.

## Правило проверяемости

Каждый пункт CLAUDE.md/AGENTS.md должен быть **проверяемым командой или явным гейтом**. «Пиши
качественно» — мусор. «Перед коммитом: `make lint && make test`» — правило. Это та же философия,
что в `harness-engineering`: enforcement > документация.

## Скелет (держи на 1–2 экрана)

```markdown
# Project — заметки для агента

## Стек и запуск
- Python 3.12, FastAPI, Postgres, Redis. Менеджер пакетов: uv.
- dev: `uvicorn app.main:app --reload`  ·  тесты: `pytest`  ·  линт: `ruff check .`
- миграции: `alembic upgrade head` (перед прод — навык migration-safety-auditor).

## Правила (нерушимые)
- Секреты только из env (`pydantic-settings`), никогда в коде/репозитории.
- Бизнес-логика — в `services/`, не в роутерах/handlers.
- Не редактировать `migrations/` вручную после применения.

## Definition of Done
- `ruff check .` и `pytest` зелёные.
- Новая публичная функция имеет docstring (Google-style).
- Архитектурное решение зафиксировано в `docs/adr/` (если нетривиально).

## Канонические доки
- Архитектура: docs/ARCHITECTURE.md   ·   Решения: docs/adr/
```

## Синхронизация CLAUDE.md и AGENTS.md

Два файла с одинаковым содержанием расходятся со временем — это **HIGH-риск** (агенты получают
разные правила). Варианты, по убыванию надёжности:

1. **Symlink (лучшее на Linux/прод):** `AGENTS.md` → симлинк на `CLAUDE.md` (или наоборот).
   Git хранит симлинки. На **Windows-dev** симлинки требуют прав/Developer Mode — учитывай.
   ```bash
   ln -s CLAUDE.md AGENTS.md      # один источник правды
   ```
2. **Один источник + тонкий указатель:** держи правила в `AGENTS.md`, а `CLAUDE.md` делает:
   ```markdown
   <!-- CLAUDE.md -->
   See AGENTS.md — single source of truth for project rules.
   ```
   (или наоборот). Минус: некоторые фичи Claude Code ждут реальный CLAUDE.md.
3. **Копия + проверка в CI/аудите:** оба файла — копии; в аудите docs-generator сверяй их
   `diff` и сообщай расхождение. Самый хрупкий вариант, но не требует симлинков.

Рекомендация под профиль (Windows-dev + Linux-прод): **вариант 2** — один содержательный файл и
тонкий указатель во втором; либо symlink, если Developer Mode включён. Зафиксируй выбор в ADR.

## Аудит agent-docs

- **CRITICAL:** CLAUDE.md/AGENTS.md указывает команду или путь, которых уже нет → агент сломает прод.
- **HIGH:** есть только один из двух файлов (второй инструмент работает без правил); файлы
  рассинхронизированы и противоречат друг другу.
- **MEDIUM:** правила непроверяемые («пиши хорошо»); раздутый файл (агент игнорирует); нет ссылок
  на canonical docs.
- **LOW:** стиль, дублирование README, устаревшие мелочи.

При запросе на серьёзную настройку (DoD с вызовами навыков, Makefile, CI, ARCHITECTURE.md) —
не дублируй `harness-engineering`, а сошлись на него или передай управление.
