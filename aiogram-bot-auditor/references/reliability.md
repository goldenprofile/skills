# Надёжность Telegram API (aiogram 3.x)

Главные причины, по которым бот «падает / тормозит / дублирует / молчит» в проде.

## Single instance — только один polling-процесс

`getUpdates` может звать **только один** процесс. Две копии (например, два запуска systemd-юнита,
или локальный запуск при работающем проде) → `TelegramConflictError` (HTTP 409), апдейты
теряются/скачут между инстансами.

- Не масштабируй polling горизонтально. Один юнит, один процесс.
- Не запускай бота локально, пока крутится прод с тем же токеном (для отладки — отдельный токен).
- Webhook так не конфликтует, но webhook-URL тоже один.

## Не блокируй event loop

aiogram 3.x полностью async. Любая блокирующая операция в хендлере (синхронный драйвер БД,
`requests`, `time.sleep`, тяжёлый CPU, синхронный файловый I/O) останавливает обработку **всех**
апдейтов, пока не завершится.

- Используй async-библиотеки (asyncpg/httpx/redis.asyncio).
- CPU-тяжёлое — в `run_in_executor`/процесс-пул; долгие задачи — в фоновую очередь, не в хендлер.
- Признак в коде: `import requests`, sync-ORM в `async def` хендлере, `time.sleep`.

## Обработка ошибок Telegram

aiogram **не** делает авто-retry по умолчанию. Ключевые исключения (`aiogram.exceptions`):

| Исключение | Когда | Что делать |
|------------|-------|------------|
| `TelegramRetryAfter` | Флуд-контроль (429), есть `.retry_after` | Подождать `retry_after` и повторить; снизить темп |
| `TelegramForbiddenError` | Пользователь заблокировал бота / кикнут | При рассылке — пометить получателя неактивным, **не** падать |
| `TelegramBadRequest` | Неверный запрос (битый markup, message not modified) | Логировать, чинить причину; часто не ретраить |
| `TelegramNetworkError` | Сеть/таймаут | Ретрай с backoff |
| `TelegramAPIError` | Базовый | Глобальный error-handler (`@dp.errors()`), не молчаливый `except: pass` |

Заведи глобальный `@dp.errors()` обработчик — иначе исключение в хендлере роняет обработку
апдейта без следа.

## Рассылки и троттлинг

Лимиты Telegram (ориентир): ~30 сообщений/сек суммарно, ~1 сообщение/сек в один чат,
~20 сообщений/мин в группу. Наивная рассылка `for user: await bot.send_message(...)` без
контроля → 429 и риск временного бана токена.

- Троттли: ограничивай темп (≤~25–30/с), делай паузы; на `TelegramRetryAfter` — sleep на
  `retry_after`.
- Лови `TelegramForbiddenError`/`TelegramBadRequest` на каждого получателя, продолжай рассылку.
- Большие рассылки выноси в фоновую задачу/очередь (не в хендлер), с возможностью паузы и
  возобновления, с учётом уже отправленных (идемпотентность).

## Таймауты

- API-вызовы: разумный таймаут сессии, чтобы зависший запрос не копился.
- Long polling: таймаут getUpdates штатный; не ставь 0.
- Внешние вызовы из хендлеров (HTTP/БД) — всегда с таймаутом, иначе один зависший запрос
  держит хендлер.

## Graceful shutdown

При рестарте (деплой, `systemctl restart`) бот должен завершаться чисто:

- `dp.run_polling` сам ловит SIGINT/SIGTERM. Но **твои** ресурсы закрывай в `dp.shutdown`-хуке:
  `bot.session.close()`, redis, пулы БД, фоновые задачи (`cancel()` + дождаться).
- `dp.startup`/`dp.shutdown` (или `@dp.startup()`/`@dp.shutdown()`) — для инициализации и
  освобождения ресурсов.
- Без этого — утечки соединений и потеря незавершённой работы при каждом деплое.

## Идемпотентность

Если бот падает/рестартует до подтверждения offset, Telegram **повторно** доставит апдейты.
Хендлеры с побочными эффектами (платежи, начисления, внешние вызовы) должны выдерживать
повторную обработку: дедуп по `update_id`/бизнес-ключу, проверка «уже сделано» перед действием.

## allowed_updates

По умолчанию `getUpdates`/webhook отдают не все типы апдейтов. Если бот использует
`callback_query`, `my_chat_member`, `chat_member`, inline и т.п. — укажи их в `allowed_updates`
(в `run_polling`/`set_webhook`), иначе апдейты не придут. aiogram умеет вычислить нужные типы
из зарегистрированных хендлеров (`dp.resolve_used_update_types()`).
