# Модели деплоя и обратная совместимость

Один и тот же DDL имеет разный риск в зависимости от того, как применяются миграции.
Это первое, что нужно установить (см. «Контекст» в SKILL.md).

## Короткий downtime (стоп → миграция → старт)

Сервис останавливается, миграция применяется, сервис запускается (`systemctl stop/start`).
Старый и новый код **не** работают одновременно.

- Обратная совместимость схемы со старым кодом **не критична** — старый код уже не запущен.
- Главный риск — **длительность блокировки**: пока идёт rewrite/долгий `CREATE INDEX`, сервис
  лежит. Оцени время на проде-объёме; для тяжёлых операций используй безопасные приёмы
  (`CONCURRENTLY`, батчи), чтобы окно простоя было предсказуемым.
- `RENAME`/`DROP` относительно безопасны (никто не обращается к схеме в момент миграции),
  но всё ещё необратимы по данным — нужен бэкап.

## Zero-downtime / rolling (старый и новый код одновременно)

Во время выката какое-то время работают **обе** версии кода против **одной** схемы. Любое
несовместимое изменение схемы ломает ещё живой старый код (или новый, если миграция отстаёт).

Правило: **схема должна быть совместима и со старым, и с новым кодом** на протяжении выката.
Отсюда — паттерн expand/contract.

### Expand / contract (на примере переименования колонки `amount` → `total`)

1. **Expand (миграция №1, до/вместе с деплоем кода):** добавить новую колонку `total`
   (nullable, без тяжёлого default). Схема совместима со старым кодом — он её игнорирует.
2. **Код:** деплой версии, которая **пишет в обе** колонки и **читает из новой** (с фолбэком
   на старую). Дождаться, пока выкат завершён везде.
3. **Backfill:** заполнить `total` из `amount` батчами для старых строк (вне транзакции).
4. **Код:** деплой версии, которая использует только `total`.
5. **Contract (миграция №2, после полного выката):** снять двойную запись, затем удалить
   `amount`. Теперь это безопасно — её никто не читает и не пишет.

Тот же приём применим к смене типа, разбиению/слиянию колонок, новым `NOT NULL`.

### Что нельзя делать одной миграцией при zero-downtime

- `RENAME` колонки/таблицы (мгновенно ломает старый код).
- `DROP COLUMN`/`DROP TABLE` до того, как код перестал к ним обращаться.
- Новый `NOT NULL`/`UNIQUE`/FK без двухфазного приёма и backfill.
- Сужение типа/ограничения, под которое не подходят существующие данные.

## «По-разному» (модель зависит от проекта)

Если модель деплоя не фиксирована, определяй её по контексту проекта (есть ли балансировщик/
несколько воркеров за nginx, делается ли graceful reload, или просто `systemctl restart`).
При сомнении — спроси и оценивай по **более строгой** (zero-downtime) модели: совместимые
миграции безопасны в обоих режимах, обратное неверно.

## SQLite и переезд на Postgres

SQLite типичен для прототипов/ранних MVP. Особенности, влияющие на миграции:

- **Урезанный `ALTER`**: поддерживает `ADD COLUMN`, `RENAME TABLE/COLUMN`, `DROP COLUMN`
  (3.35+). Смена типа и большинство constraint невозможны напрямую — Django/Alembic
  пересоздают таблицу (table rebuild): медленно и блокирующе, а писатель только один.
- **Слабая типизация и неэнфорсимые constraint** (FK по умолчанию выключены): данные могут
  нарушать ограничения, которые Postgres применит строго.

Сигналы «пора на Postgres» (мигрировать **до** роста данных/нагрузки): конкурентная запись,
таблицы, на которых table-rebuild занимает заметное время, потребность в строгих constraint,
частичных индексах, `CONCURRENTLY`. При переезде проверь данные на нарушения будущих
constraint (NULL там, где станет `NOT NULL`; дубликаты под будущий `UNIQUE`; «висячие» FK)
до применения строгой схемы.
