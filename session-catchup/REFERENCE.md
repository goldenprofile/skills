# Session Catchup — справка

Детали по сбору состояния и решению пограничных ситуаций. Основная инструкция — в [SKILL.md](SKILL.md).

## Источники состояния: приоритет и что извлекать

Источники упорядочены по достоверности. Git — самый надёжный (отражает реальное состояние кода); история диалога — самый полный по намерению, но может быть потерян/сжат.

| Источник | Что извлекать | Команды / где смотреть |
|----------|---------------|------------------------|
| git working tree | что реально изменено сейчас | `git status`, `git diff --stat`, `git diff` |
| git stash | отложенная работа (легко забыть) | `git stash list` |
| git history | что уже завершено и закоммичено | `git log --oneline -15`, `git log --stat -3` |
| текущая ветка | контекст фичи/задачи | `git branch --show-current` |
| история диалога | намерение, на чём прервались | прокрутка контекста, summary компактификации |
| TODO/план сессии | заявленные шаги и их статус | TodoWrite-список, если велся |
| файлы состояния | сохранённый ранее контекст | `.claude/`, `docs/`, notepad/state плагинов |
| канонические доки | правила проекта, последние изменения | `CLAUDE.md`, `ARCHITECTURE.md`, `CHANGELOG.md`, `ROADMAP.md` |

### Сопоставление намерения и факта

Ключевой приём: сравни **что собирались делать** (история/TODO) с **что реально в git**. Расхождения — это и есть «не завершено»:

- в истории обсуждали фичу X, но в `git diff` её нет → не начато или потеряно;
- в `git diff` есть изменения, которых не было в плане → побочная работа, зафиксируй её;
- коммит есть, но тесты на него не гонялись → кандидат на диагностику.

## Поиск команд проекта (не угадывать)

Извлекай реальные команды, а не предполагай:

- **Makefile** — цели `test`, `lint`, `build`, `check` (`grep -E '^[a-z-]+:' Makefile`).
- **package.json** — секция `scripts`.
- **pyproject.toml / setup.cfg** — секции инструментов (ruff, pytest, mypy).
- **CLAUDE.md / README** — раздел «команды»/«частые команды».
- **CI-конфиги** — `.github/workflows/*.yml`, `.gitlab-ci.yml` — что гоняется на CI.

Если нашёл несколько вариантов — предпочитай агрегирующую цель (`make check`), затем отдельные (`make test`, `make lint`).

## Red flags при беглом просмотре изменённого

Смотри только в файлах из `git diff` (не весь проект):

- `TODO` / `FIXME` / `XXX` / `HACK`, добавленные в этой сессии;
- отладочные артефакты: `print(`, `console.log(`, `debugger`, `breakpoint()`, `dd(`, `var_dump(`;
- оборванные/неиспользуемые импорты;
- закомментированные блоки кода вместо удаления;
- секреты/токены/пароли в открытом виде (если есть — это блокер, в начало сводки).

## Расширенная диагностика (только по запросу)

Если беглая проверка выявила системные проблемы, предложи (но не запускай сам) специализированный инструмент:

- общий аудит проекта → `python-project-audit` / `code-review`;
- архитектура → `django-architect` / `improve-codebase-architecture`;
- безопасность → `django-security-shepherd` / `security-review`;
- мёртвый код → `django-cleanup-inspector` / `refactor-clean`;
- готовность к деплою → `django-deploy-validator`.

Выбор зависит от стека проекта — сверься с `CLAUDE.md`.

## Что НЕ делает этот навык

- Не сохраняет сессию — для этого `save-session`.
- Не готовит передачу другому агенту — для этого `handoff`.
- Не делает глубокий разбор незнакомого проекта с нуля — для этого `code-archaeologist`.
- Ничего не правит, не коммитит, не пушит, не запускает миграции/деструктивные команды.
