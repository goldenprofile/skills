# Ручной review — 8 испытаний

Оцени каждую категорию по шкале 0-10. Для каждой находки: файл, строка,
проблема, решение.

## Испытание 1: Архитектура (0-10)

- Чёткое разделение слоёв (routes / services / repositories / models)
- Отсутствие бизнес-логики в route-хендлерах
- Dependency injection (особенно FastAPI Depends)
- Конфигурация вынесена из кода (pydantic-settings / django-environ)
- Нет циклических импортов
- Единая точка входа, понятная структура пакетов

**Антипаттерны:** God-objects, спагетти-импорты, логика в `__init__.py`,
хардкод URL/credentials.

## Испытание 2: Обработка ошибок (0-10)

- Кастомные exception-классы для доменных ошибок
- Глобальный error handler (middleware)
- Нет голых `except:` или `except Exception:`
- HTTP-ответы с правильными статус-кодами
- Логирование ошибок с контекстом (не `print()`)
- Graceful degradation при недоступности внешних сервисов

**Антипаттерны:** `pass` в except, проглатывание ошибок, 500 на всё,
трейсбеки в ответах API.

## Испытание 3: Безопасность (0-10)

- Валидация входных данных (Pydantic / Django forms / marshmallow)
- SQL-injection защита (ORM, параметризованные запросы)
- Аутентификация и авторизация (JWT, OAuth2, RBAC)
- CORS правильно настроен (не `allow_origins=["*"]` в проде)
- Секреты в env-переменных, не в коде
- Rate limiting
- HTTPS-only в конфигурации прода
- Защита от SSRF, XSS, CSRF (где применимо)

**Антипаттерны:** секреты в git, `DEBUG=True` в проде, отключённая валидация,
`eval()`/`exec()`.

## Испытание 4: Производительность и Highload (0-10)

- Async/await для I/O (FastAPI/Starlette)
- Connection pooling для БД
- Индексы на часто используемых полях
- N+1 запросы отсутствуют (select_related / joinedload)
- Кэширование (Redis/Memcached) где уместно
- Пагинация на всех list-эндпоинтах
- Background tasks для тяжёлых операций (Celery/ARQ/BackgroundTasks)
- Streaming для больших ответов

**Антипаттерны:** синхронный код в async эндпоинтах, загрузка всей таблицы
в память, отсутствие лимитов на размер запроса.

## Испытание 5: Тестирование (0-10)

- Покрытие > 70% — хорошо, > 85% — отлично
- Юнит-тесты для бизнес-логики
- Интеграционные тесты для API-эндпоинтов
- Фикстуры и фабрики (factory_boy / faker)
- Тесты на edge-cases и ошибки
- Нет зависимости от внешних сервисов в тестах (моки)

**Антипаттерны:** тесты с `assert True`, тесты зависят от порядка,
нет `conftest.py`, тесты ходят в реальную БД.

Для глубокой проверки качества тестов используй навык **test-coverage-auditor**.

## Испытание 6: Инфраструктура и деплой (0-10)

- Dockerfile есть и оптимизирован (multi-stage, .dockerignore)
- docker-compose для локальной разработки
- CI/CD pipeline настроен
- Миграции БД управляемые (Alembic / Django migrations)
- Health-check эндпоинт
- Graceful shutdown
- Environment-based конфигурация (dev/staging/prod)

**Антипаттерны:** `pip install` в CMD, root-пользователь в контейнере,
нет .dockerignore, миграции вручную.

## Испытание 7: Качество кода и стиль (0-10)

- Единый стиль (black/ruff formatted)
- Type hints на публичных функциях
- Docstrings на сложной логике
- Нет магических чисел/строк (constants/enums)
- Файлы < 300 строк (в среднем)
- Функции < 30 строк (в среднем)
- Понятные имена переменных и функций

**Антипаттерны:** `x`, `data`, `result` без контекста, файлы на 1000+ строк, копипаста.

## Испытание 8: Документация и DX (0-10)

- README с инструкцией по запуску
- API-документация (OpenAPI/Swagger авто или вручную)
- .env.example с описанием переменных
- CONTRIBUTING.md или аналог
- Changelog ведётся
- Комментарии объясняют «почему», а не «что»

**Антипаттерны:** «запусти и разберёшься», недокументированные env-переменные,
устаревший README.
